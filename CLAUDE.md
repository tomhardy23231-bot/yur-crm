## 1. О проекте

CRM-система для юридической компании.

Центральная сущность — **«Дело»** (оно же договор; отдельной сущности «договор» НЕТ).
Вокруг дела собираются: клиент, документы, задачи и сроки, команда и финансы.

Система состоит из:
- **внутреннего интерфейса** для сотрудников (основное);
- **клиентского портала** (позже, Phase 3) — клиент видит статус своего дела и документы.

Язык интерфейса — **русский**. Идентификаторы в коде и БД — **английские**, с понятными именами.

## 2. Технологический стек

| Слой | Выбор |
|---|---|
| Frontend + backend | **Next.js (App Router) + TypeScript** |
| БД / Auth / Storage | **Supabase** (PostgreSQL + Auth + Storage + Row Level Security) |
| Доступ к данным | `supabase-js` на сервере, **с сессией пользователя** (чтобы работал RLS) |
| Схема и миграции | SQL-миграции через **Supabase CLI** (`supabase/migrations`) |
| UI | **Tailwind CSS + shadcn/ui** |
| Хранение файлов по делам | **Supabase Storage** (приватный бакет) |
| Деплой | Vercel (или любой Node-хостинг) |

**Почему так.** RLS в Postgres напрямую ложится на нашу модель доступа («специалист
видит только свои дела»): правило проверяется в самой базе, а не только в коде —
это надёжнее и безопаснее. Один Next.js-проект закрывает и внутренний интерфейс,
и будущий клиентский портал через ролевые маршруты.

**Важно про RLS (легко сломать):** запросы к данным от лица пользователя должны идти
через клиент Supabase **с его JWT/сессией**, иначе RLS не сработает. `service_role`
(ключ в обход RLS) использовать только для системных задач (миграции, сидинг,
фоновые операции) — никогда для обычных пользовательских запросов.

**Стек можно поменять.** При смене правится только этот раздел и раздел «Команды»;
доменная модель и бизнес-правила (разделы 4–8) не меняются.

```bash

npx supabase stop             # остановить
npx supabase status           # URL и ключи локального стека → копировать в .env.local
npx supabase db reset         # пересоздать БД из миграций + seed.sql
npx supabase migration new <name>   # создать новый файл миграции
npx supabase db push          # применить миграции на удалённый Supabase (когда подключим)
```

Studio (веб-интерфейс БД) после `supabase start` доступен на `http://127.0.0.1:54323`.

### Seed (тестовые данные)
```bash
npm run db:seed   # появится в Шаге 1; использует SUPABASE_SERVICE_ROLE_KEY
```

## 4. Роли и доступ

Четыре роли (`users.role`): `owner`, `admin`, `specialist`, `assistant`.

- **owner** (владелец / супер-админ) — полный доступ ко всему: настройки системы,
  управление пользователями и правами, все дела и финансы.
- **admin** (офис-менеджер / секретарь) — заводит клиентов и дела, прикрепляет
  документы, следит за порядком. Видит **все** дела, клиентов и **финансы**.
  **НЕ** управляет ролями и системными настройками (это только owner).
- **specialist** (адвокат / юрист) — ведёт свои дела. Адвокат и юрист имеют
  **одинаковые** права, различаются только полем `specialist_type` (`lawyer` | `jurist`).
- **assistant** (помощник) — закреплён за **одним** специалистом
  (`users.supervisor_id`). Видит дела своего специалиста.

> ⚠️ Изменение относительно исходной концепции: помощник привязан к ОДНОМУ специалисту,
> а не назначается в «команду дела». Причина (ответ клиента): большинство специалистов
> внештатные и работают сами на себя, у каждого свой помощник.

### Матрица доступа

| Действие | owner | admin | specialist | assistant |
|---|---|---|---|---|
| Свои дела | ✅ | ✅ (все) | ✅ (только свои) | ✅ (дела своего специалиста) |
| Чужие дела специалистов | ✅ | ✅ | ❌ | ❌ |
| Финансы по делам | ✅ | ✅ | ✅ (по своим) | ✅ (по делам специалиста) |
| Клиенты | ✅ | ✅ | свои | дела специалиста |
| Управление пользователями/ролями | ✅ | ❌ | ❌ | ❌ |
| Системные настройки | ✅ | ❌ | ❌ | ❌ |

**Правила RLS (для cases и связанных таблиц):**
- `owner`, `admin` → все строки;
- `specialist` → `cases.responsible_id = auth.uid()`;
- `assistant` → `cases.responsible_id = (supervisor_id текущего пользователя)`;
- документы/задачи/платежи наследуют доступ от своего дела.

## 5. Доменная модель

> Имена таблиц/полей — английские. Глоссы в скобках — для понимания домена.

**users** — сотрудники
`id, full_name, email, role (owner|admin|specialist|assistant), specialist_type (lawyer|jurist|null), supervisor_id (для assistant → specialist), is_active, created_at`

**clients** (доверители)
`id, name, client_kind (individual|company), phone, email, address, notes, created_by, created_at`
— у одного клиента может быть несколько дел.

**cases** (дела; оно же договор — центральная сущность)
`id, number_title (обязат.), client_id (обязат.), responsible_id (обязат., specialist),
opened_at (обязат.), case_type (обязат.: civil|criminal|corporate|...), stage (обязат., см. §6),
priority (normal|urgent), tags (text[]),
contract_sum, paid_total, debt, billing_types (text[]: prepaid|hourly|fixed|success_fee),
opponent, court_case_number, court, closed_at, created_at`
— обязательные поля помечены; остальные опциональны (суд — только если дошло до суда и т.д.).

**documents**
`id, case_id, file_name, storage_key, doc_type (contract|claim|power_of_attorney|correspondence|other), uploaded_by, uploaded_at`

**tasks** (задачи и сроки)
`id, case_id, title, description, kind (task|hearing|deadline), assignee_id, created_by, due_at, status (open|done), created_at`
— любой специалист может ставить задачи себе и коллегам. Питает общий календарь.

**payments** (оплаты)
`id, case_id, amount, paid_at, method, note, created_by`
— `paid_total` и `debt` в case считаются из платежей и `contract_sum`.

**activity_log** (история изменений)
`id, entity_type, entity_id, user_id, action, changes (jsonb), created_at`

**Phase 2/3 (проектируем позже):** `document_templates`, `invoices`, `time_entries`,
`client_portal_users`, `client_requests` (обращения с портала).

## 6. Этапы дела (воронка)

Линейная воронка, движение **только вперёд** (см. §7):

1. `new_request` — Новое обращение
2. `consultation` — Консультация и оценка
3. `in_progress` — Договор заключён, в работе
4. `pretrial` — Досудебное урегулирование
5. `litigation` — Судебное производство
6. `awaiting_decision` — Ожидание решения
7. `enforcement` — Исполнение решения
8. `closed` — Завершено (в архив)

> Воронку клиент подтвердил «пока так, потом подправим» — заложи `stage` как enum,
> который легко расширять/менять.

## 7. Бизнес-правила (соблюдать строго)

1. **Договор = Дело.** Отдельной сущности «договор» нет.
2. **Этапы — только вперёд.** Возврат на предыдущий этап в обычной работе запрещён
   (подтверждено клиентом). Исключение: ручное исправление ошибочно выставленного
   этапа — только `owner`/`admin`, с записью в `activity_log`.
3. **Видимость дел:** специалист видит только свои; адвокаты и юристы НЕ видят дела
   друг друга. Помощник видит дела своего специалиста.
4. **admin видит финансы** по всем делам, но НЕ управляет ролями/настройками.
5. **Задачи** может ставить любой специалист — себе и коллегам.
6. **Источник клиента НЕ фиксируем** (поле не нужно — ответ 17).
7. Все изменения по делам пишутся в `activity_log` (кто, что, когда).

## 8. Объём работ по фазам

**Phase 1 — MVP (делаем сейчас):**
- Auth + роли + RLS (модель доступа из §4);
- Клиенты (CRUD, список их дел);
- Дела + карточка дела (все поля из §5);
- Этапы: доска (kanban) и/или список, движение только вперёд;
- Задачи + общий календарь заседаний/дедлайнов;
- Документы: загрузка/хранение/скачивание по делу;
- Финансы (базово, вручную): `contract_sum`, платежи, расчёт долга;
- Поиск и фильтры (по статусу, типу, клиенту, ответственному);
- Журнал изменений (`activity_log`);
- Напоминания о приближающихся сроках/заседаниях (хотя бы в интерфейсе).

**Phase 2:**
- Шаблоны документов с автоподстановкой данных дела (договор, претензия, доверенность);
- Выставление счетов (инвойсы) из системы;
- Учёт рабочего времени (`time_entries`) — для почасовой оплаты и аналитики;
- Расширенные отчёты и аналитика.

**Phase 3:**
- Клиентский портал (отдельный интерфейс: статус дела, документы, обращения);
- Электронная подпись документов;
- Внешние интеграции (почта, мессенджеры, бухгалтерия, судебные реестры);
- Миграция существующих данных.

## 11. Дизайн интерфейса

> **Источник правды — `DESIGN.md` в корне проекта.** Любой UI-код (шрифт, цвет,
> отступ, радиус, компонент) читает `DESIGN.md` ПРЕЖДЕ, чем выбрать значение.
> Отклонения — только с явным согласованием. Ниже — короткие принципы; всё
> остальное (токены, шкалы, правила, запреты) — в `DESIGN.md`.

### Характер
- **Архетип:** NetHunt CRM × Stripe Dashboard. Современный prosumer SaaS, яркий,
  цветной, плотный data-tool. Не refined-minimal, не «1С-look».
- **Тон:** светлый, оптимистичный, цветные визуальные якоря (этап / приоритет /
  источник коммуникации) как основной язык. Бренд — индиго.
- **Тёмная тема в Phase 1 НЕ делается.** Только светлая.

### Жёсткие запреты
- НЕ Inter, Roboto, Arial, system-ui для основных ролей.
- НЕ serif в display (Lora, Newsreader, Fraunces — мимо).
- НЕ градиенты на кнопках или больших фонах. Градиенты — только в hero-шапках
  карточек (контакт, лид, заявка).
- НЕ дефолтный shadcn-look (slate-палитра, neutral).
- НЕ «1С-look»: тёмная панель навигации, плотные модалки с табами сверху, серый dashboard.

### Типографика
- **Plus Jakarta Sans** (UI/display) + **Geist Mono** (numbers). Кириллица обязательна.
- `font-variant-numeric: tabular-nums` глобально.

### Цвет и токены
- Все цвета через CSS-переменные / Tailwind 4 `@theme inline`. Никаких хардкодов.
- Один primary (индиго `#4F46E5`) + семантика + 8 цветов этапов + 3 цвета приоритетов.

### Плотность и макет (data-tool)
- Comfortable density для карточек/форм, плотные таблицы (44px ряд, sticky headers).
- Карточка дела — главный экран: секции (данные · документы · задачи · финансы · история).
- Командная палитра (Cmd/Ctrl-K) и горячие клавиши — заложить с начала.

### Движение
- Primary-кнопка на hover — мягкий lift + indigo-shadow (NetHunt-style).
- Сдержанно остальное. Скорость важнее эффектов.

### Состояния
- Empty-state, skeleton, error, confirmation — всегда продуманы.

### Компоненты
- Архитектура shadcn/ui (CVA + Radix + Tailwind), но имена токенов наши.
  Компоненты пишутся сразу под наши токены, не копируются дефолтные.

### Контроль качества
- Перед версткой нового экрана → читать `DESIGN.md`.
- После верстки → `/design-review` (полировка) + `/qa` (поведение в браузере).
- Доступность (контраст AA, фокус, клавиатура) обязательна.
