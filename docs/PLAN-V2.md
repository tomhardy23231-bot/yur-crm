## Протокол сессии «ПРОДОЛЖАЙ»

Пользователь начинает сессию словом **«ПРОДОЛЖАЙ»**. Агент обязан:

1. Прочитать `CLAUDE.md` (особенно §4, §5, §7), затем этот файл: **статус-таблицу**
   ниже и раздел своего этапа, затем последнюю запись `docs/PROGRESS.md`.
2. Определить по статус-таблице первый этап со статусом ⬜ — это и есть задача сессии.
   Делать **только его**. Следующий этап в той же сессии НЕ начинать.
3. Перед миграциями/крупными решениями — короткий план пользователю и ждать «ок»
   (правило проекта).
4. **Завершение этапа — обязательный чеклист:**
   - [ ] `npx tsc --noEmit` + `npm run lint` + `npm run build` — зелёные;
   - [ ] миграции применяются на чистую БД: `npx supabase db reset`;
   - [ ] тесты затронутых областей проходят (Vitest; для RLS — integration-тесты видимости);
   - [ ] `/review` диффа; для этапов с RLS/правами (1, 2, 4, 5, 7) — дополнительно `/cso`;
   - [ ] обновить разделы `CLAUDE.md`, указанные в этапе («Доки» в конце этапа);
   - [ ] отметить этап в статус-таблице ниже (✅ + дата + хеш коммита);
   - [ ] добавить handoff-запись в `docs/PROGRESS.md`;
   - [ ] один общий коммит этапа (не дробить); **пуш на прод (master → Vercel) — только
     после явного «ок» пользователя**;
   - [ ] деплой миграций на прод: `npx supabase db push` — тоже только по «ок»
     (⚠ помнить гочу allowlist `activity_log`, см. «Подводные камни»).

## Целевая модель (спека, согласована)

### Подразделения и доступ
- Новая сущность `departments` — изначально 10 (Київський, Дніпровський, Львівський,
  Одеський + «Підрозділ 5…10» — названия клиент даст позже). Количество не ограничено.
- Каждый сотрудник привязан к подразделению (`users.department_id`) и имеет
  отображаемую должность (`users.position`, свободный текст: керівник, заступник,
  юрист ВП, менеджер ВП, експерт, адміністратор). Штатка клиента ~20 чел/подразделение,
  лимитов нет.
- **Роли доступа остаются прежние 5** (owner, admin, office_manager, lawyer, expert).
  Маппинг должностей: керівник/заступник → `admin`; юрист ВП/менеджер ВП → `lawyer`;
  експерт → `expert`; адміністратор → `office_manager`.
- Дела пересекают подразделения: дело «принадлежит» подразделению юриста-продажника
  И подразделению эксперта-исполнителя одновременно — его видят оба руководителя.
- **Видимость:**
  - `owner` — всё, всегда (режим бога), не настраивается;
  - `admin`/`office_manager` — `users.visibility_scope`: `'department'` (по умолчанию —
    видит дела/финансы/людей только своего подразделения) или `'all'` (вся компания).
    Выставляет только owner, индивидуально каждому сотруднику (из карточки подразделения);
  - `lawyer`/`expert` — всегда только свои дела (`lawyer_id`/`responsible_id`),
    НЕ настраивается;
  - переходное правило: admin/office_manager с `department_id IS NULL` видит всё
    (поведение как до v2), чтобы система не «ослепла» до раскидывания людей.

### Зарплата — три режима (на сотруднике)
- `users.salary_mode`: `'percent'` (по умолчанию, текущая модель) | `'fixed'` |
  `'fixed_percent'`; `users.salary_fixed_amount` — фикс в месяц.
- Процентная часть — существующая механика без изменений (ставка категории
  `payroll_rates` + override на деле).
- Права назначения: owner — всем; admin — сотрудникам **своего** подразделения.

### Акты = платёжные документы (мини-биллинг)
Документ — **«Рахунок-Акт»** (счёт и акт в одном, образец клиента:
`docs/samples/rahunok-akt-sample.xlsx`). Это же закрывает тему «инвойсов» из Phase 2.

Цикл (уточнён с клиентом 10.06.2026):
1. **СРМ генерирует акт** по образцу: сквозной номер («Рахунок-Акт № N від <дата>»),
   реквизиты исполнителя из настроек компании, заказчик из карточки клиента
   (ФИО/назва + РНОКПП/ЄДРПОУ), услуга и сумма, сумма прописью (укр.),
   призначення платежу, блок «Умови прийняття», подписи.
2. Акт выдаётся клиенту (печать / PDF / отправка) — статус `issued`.
3. Клиент оплачивает.
4. В СРМ **загружают скан с подписями/печатью и прописывают оплаченную сумму** —
   статус `paid`: автоматически создаётся платёж по делу на эту сумму; отметка
   выполнения `full`/`partial` вычисляется по сумме (накопительно оплаченное по
   актам ≥ суммы договора → `full`, иначе `partial`; staff может поправить вручную).
   Скан обязателен — именно подписанный экземпляр считается подтверждением.
5. После подтверждения пересчитываются `paid_total`/долг и **просчитываются выплаты
   (ЗП)** существующим движком от оплат. Предоплаты вносятся платежами напрямую.

- Закрытие дела: **блок**, если нет акта `completion='full'` со статусом `paid`;
  совсем без актов — мягкое предупреждение (как было). ⚠ Допущение — уточняется у
  клиента, см. «Открытые вопросы».
- Эксперт получает право создавать задачи по своим делам (правка §7-6).

### Отпуска
`absences`: тип (отпуск/болезнь/другое), период с…по, комментарий. Вносит сам сотрудник,
admin своего подразделения, owner. Видно в карточке сотрудника и в календаре.

### Касса (по образцу XLS ОЛІМП)
- Счета: Карта / Рахунок / Готівка (+добавляемые), у счёта начальный остаток и дата.
- Журнал операций: приход/расход за день, свободное описание (аренда, налоги, реклама —
  НЕ привязаны к делам); платежи по делам автоматом падают приходом на выбранный счёт.
- Отчёт: разворот по дням (остаток на начало → приходы (до N строк) → расходы →
  остаток на конец), вкладки по счетам + сводный Total; сальдо на день и месяц.
- Доступ: по умолчанию только owner; owner выдаёт право точечно.

### Открытые вопросы (не блокируют этапы 1–4)
1. Точные условия блокировки закрытия дела: частичный акт — блок (решено);
   полный, но неоплаченный акт — блок? (принято допущение «да»). Подтвердить у клиента.
2. Названия остальных 6 подразделений — клиент даст позже (заводим «Підрозділ N»).
3. Порог `full`/`partial`: принято допущение «оплачено по актам ≥ суммы договора →
   full» (+ ручная правка staff). Подтвердить у клиента на этапе 5.

## Этап 2 — RLS: новая модель доступа

**Цель:** переключить видимость с «staff видит всё» на модель подразделений.

Задачи:
- SQL-функции: `current_user_department()`; переписать `can_see_case` /
  `can_write_case` (и места, где используется `is_staff()` для видимости данных):
  - owner → всё;
  - admin/office_manager: `visibility_scope='all'` ИЛИ `department_id IS NULL` → всё;
    иначе → дела, где подразделение `lawyer_id` ИЛИ `responsible_id` совпадает с их;
  - lawyer/expert — без изменений.
- `is_staff()` оставить для операций, не зависящих от подразделения (создание
  клиентов/дел и т.п.); деструктив (`can_manage_users`) и системные настройки
  (`is_owner`) — без изменений.
- Наследуемые таблицы (documents, tasks, payments, comments, activity_log по делу)
  автоматически следуют новому `can_see_case` — проверить каждую политику.
- Отчёты (payroll и пр.): admin видит людей/цифры своего подразделения.
- Integration-тесты видимости: матрица «руководитель Киева видит дело Киев-продал/
  Днепр-исполняет; руководитель Львова — НЕ видит; scope='all' видит всё; NULL
  department видит всё; lawyer/expert — без изменений».

DoD: тесты матрицы зелёные; `/cso` + `/review` пройдены.
Доки: CLAUDE.md §4 (матрица доступа), §7-3, §7-5.

## Этап 5 — Акты как платёжные документы

**Цель:** цикл «сгенерирован → выдан → оплачен (скан + сумма) → платёж → ЗП».
**Образец документа:** `docs/samples/rahunok-akt-sample.xlsx` («Рахунок-Акт № 40»).

Задачи:
- **Предпосылки для генерации:**
  - реквизиты компании-исполнителя в настройках (owner): назва, ЄДРПОУ, адреса,
    телефон, IBAN + банк/МФО, строки налогового статуса («Не є платником ПДВ»,
    «Є платником єдиного податку, 3 група») — таблица `org_requisites` или
    JSON-настройка; решить на этапе;
  - `clients.tax_id text null` — РНОКПП/ЄДРПОУ заказчика (+ поле в форме клиента);
  - утилита «сумма прописью» на украинском («Дев'ятнадцять тисяч гривень 00 копійок»).
- Миграция: `case_acts` (`id`, `case_id not null`, `number int not null` — сквозная
  нумерация (sequence), `amount numeric not null check (> 0)`,
  `confirmed_amount numeric null` — прописанная при подтверждении сумма,
  `completion text null check (completion in ('full','partial'))`,
  `status text not null default 'issued' check (status in ('issued','paid'))`,
  `issued_at date not null`, `paid_at date null`,
  `scan_document_id uuid null references documents(id)` — подписанный скан,
  `created_by`, `created_at`).
- **Генерация печатной формы** по образцу: print-friendly страница → PDF через печать
  браузера ИЛИ exceljs-шаблон XLSX — выбрать на этапе (критерий: 1:1 с образцом,
  без тяжёлых зависимостей). Номер и дата — автоматически, шапка/подвал из реквизитов.
- RLS: видимость — наследование от дела (`can_see_case`); создание — expert своего
  дела + staff с доступом к делу; подтверждение оплаты — lawyer своего дела +
  admin/owner (создаёт payment → правка/удаление самих payments остаётся owner/admin).
- Подтверждение «оплачен»: форма «скан (обязателен) + оплаченная сумма + дата» →
  автосоздание `payments` (сумма = confirmed_amount, связь `payments.act_id`) →
  существующие триггеры paid_total/debt/ЗП работают; `completion` вычисляется
  (сумма оплат по актам ≥ `contract_sum` → `full`, иначе `partial`), staff может
  переопределить вручную.
- Блок закрытия дела: переход в `closed` запрещён, если нет акта `full`+`paid`
  (БД-триггер + понятная ошибка в UI); актов нет вообще — мягкое предупреждение
  (текущее поведение сохраняется).
- Право эксперта создавать задачи по своим делам (политика tasks + UI).
- Карточка дела: секция «Акты» (сгенерировать, скачать/печать, подтвердить оплату;
  статусные бейджи issued/paid + full/partial), оптимистичный UI по образцу платежей.
- `activity_log`: новые действия по актам (⚠ allowlist!).

DoD: e2e-флоу «акт сгенерирован (номер/реквизиты/сумма прописью корректны) → скан +
сумма → платёж появился → долг уменьшился → ЗП выросла → дело закрывается только при
full+paid» покрыт integration-тестом; печатная форма сверена с образцом глазами;
`/cso` + `/review`.
Доки: CLAUDE.md §5 (case_acts, clients.tax_id, org_requisites), §6 (закрытие),

## Этап 7 — Касса и сальдо-отчёт

**Образец:** `docs/samples/oborotka-olimp-sample.xls` (3 листа-кассы: по дню — остаток

Задачи:
- Миграции: `cash_accounts` (`id`, `name`, `kind text check (in 'card','bank','cash')`,
  `opening_balance numeric not null default 0`, `opening_date date not null`,
  `is_active bool default true`); `cash_entries` (`id`, `account_id not null`,
  `entry_date date not null`, `direction text check (in 'in','out')`,
  `amount numeric check (> 0)`, `description text not null`, `case_id uuid null`,
  `payment_id uuid null`, `created_by`, `created_at`).
- Право: `users.can_manage_cash boolean default false` — выдаёт только owner
  (owner всегда имеет доступ).
- RLS: всё по `can_manage_cash` или owner.
- Автоприход: при внесении платежа по делу — выбор счёта (маппинг `payments.method` →
  счёт по умолчанию), создаётся `cash_entries(in)` со ссылкой на платёж; правка/откат
  платежа корректно отражается в кассе.
- Сидинг: 3 счёта (Карта, Рахунок, Готівка).
- UI `/reports/cash`: вкладки счетов + Total; таблица по дням с разворотом
  (остаток на начало → приходы → расходы → остаток на конец); итоги месяца;
  ручное добавление операции (приход/расход, счёт, дата, сумма, описание);
  моноширинные цифры, минус — красным.
- Сальдо считается накопительно от `opening_balance`/`opening_date` (выводить
  предупреждение, если есть операции раньше opening_date).

DoD: суммы сходятся с контрольным примером из XLS (юнит-тест на расчёт сальдо);
`/cso` (новый модуль прав) + `/review`. Доки: CLAUDE.md §5, §8 (Phase 2 — касса
реализована).

## Подводные камни (для всех этапов)

- **RLS:** запросы от лица пользователя — только через клиент с его сессией;
