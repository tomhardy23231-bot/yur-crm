## 1. Мотивация (со слов владельца, 2026-07-14)

1. **Тормозит на проде** — сессия проверяется сетевым запросом к Auth-серверу
   Supabase на каждый заход (фикс getClaims в коде есть, но на проде не
   активирован: нужны асимметричные ключи; владелец решил не продолжать
   инвестировать в Supabase).
2. **Морока с локальной разработкой** — Docker + supabase CLI + локальный стек.
3. **Сам сервис неудобен** — хочется «как в BudovaCRM»: Neon-консоль,
   ветки БД, никакой обвязки.

Решение владельца: полный уход с Supabase на чистый Postgres. Требование —
**без потерь**: данные, пароли сотрудников, файлы, вся функциональность.

4. **Конечная цель (вводная 2026-07-14): корпоративный сервер.** Под CRM
   позже купят собственный сервер компании — система должна переехать на него
   малой кровью. Поэтому ВСЕ выборы этого цикла делаются без vendor lock-in:
   - БД: обычный Postgres (Neon сейчас → свой сервер потом = pg_dump + смена
     `DATABASE_URL`);
   - файлы: S3-совместимый интерфейс (R2 сейчас → MinIO или диск на корп-сервере,
     реализация — один модуль за интерфейсом `lib/storage.ts`);
   - почта: nodemailer + SMTP-конфиг (Resend SMTP сейчас → корпоративный SMTP потом,
     без правки кода);
   - auth: свой, в приложении — переносится вместе с ним;
   - приложение: Vercel сейчас → Docker/standalone-сборка Next.js на корп-сервере
     потом (отдельный будущий цикл, не в v4).

## 2. Что НЕ меняется

- Весь UI, все экраны, вся бизнес-логика и правила из CLAUDE.md §4–§7.
- **Вся схема БД и все 70 миграций** — это уже чистый Postgres-SQL.
  RLS-модель доступа СОХРАНЯЕТСЯ (см. §4.1) — не переписываем безопасность в коде.
- Все ~26 SQL-функций (confirm_act_paid, payroll_*, search_case_ids, cash_*,
  dashboard_*, user_login_secrets-механика…) — работают как есть.
- Пароли сотрудников — bcrypt-хеши из `auth.users` переносятся, все входят
  со старыми паролями.
- Деплой на Vercel, Telegram-бот, ICS-календарь, OnlyOffice, Vercel Cron.
- Механика «владелец выдаёт логин+пароль» (`user_login_secrets` + модалка).

## 3. Целевая архитектура — таблица замен

| Сейчас (Supabase) | Станет | Комментарий |
|---|---|---|
| Postgres в Supabase | **Neon** (регион Frankfurt = регион Vercel-функций) | ветки production + development, как в BudovaCRM |
| Supabase Auth (GoTrue, сетевой) | **Свой auth**: JWT в httpOnly-cookie (`jose`, HS256) + refresh-токены в таблице `auth.sessions`, пароли `bcryptjs` | проверка сессии ЛОКАЛЬНАЯ (микросекунды) — главный убийца тормозов |

| supabase-js (PostgREST) | **Prisma** (клиент, introspection из готовой схемы) + `$queryRaw` для RPC | тот же стек, что BudovaCRM; embedded-join → include |

| Supabase Storage (бакет case-documents) | **Cloudflare R2** (S3-совместимый, приватный; 10 ГБ бесплатно) | интерфейс `lib/storage.ts`: upload / download / signedUrl / remove — 4 операции в ~6 местах; на корп-сервере реализация меняется на MinIO/диск без правки вызовов |
| Приглашения/сброс пароля по email (Supabase mailer) | **Своя отправка** (решение владельца): nodemailer + SMTP (Resend SMTP), свои шаблоны (укр., «ЮрКейс») — **сессия 8, ПОСЛЕ переезда прода** (ревью D1: не добавлять новые подсистемы в момент переезда) | до сессии 8 выдача доступов — копи-блоком «логин+пароль» (основной флоу и сейчас); на корп-сервере SMTP меняется env-переменными |
| supabase CLI + Docker локально | **ничего**: dev-ветка Neon + `npm run db:migrate` (свой раннер SQL-миграций ~50 строк) | локалка = просто `npm run dev` |
| Supabase Studio | Neon Console + psql | привычно по BudovaCRM |
| Бэкапы: НЕТ (free tier) | Neon point-in-time restore (**6 часов** на free — ревью V3-факт; больше на платном) + наши дампы в `/backups/` остаются ОБЯЗАТЕЛЬНЫМИ | лучше, чем сейчас, но дампы не отменяет |

Новые зависимости: `prisma`, `@prisma/client`, `pg`, `jose`, `bcryptjs`,
`@aws-sdk/client-s3` (если R2). Удаляются: `@supabase/ssr`, `@supabase/supabase-js`.

## 4. Ключевые технические решения

### 4.1 RLS сохраняем через шим-миграцию `0000_shim.sql`

Вся модель доступа (§4 CLAUDE.md: case_visible, can(), payroll_user_visible,
12 прав, гарды salary/visibility/rate_overrides) живёт в БД и завязана на
`auth.uid()` и роли `authenticated`/`service_role`/`anon`. Вместо переписывания
в TS (огромный объём + риск дыр в юр-данных) создаём шим ДО прогона миграций:

- `CREATE ROLE authenticated NOLOGIN`, `anon NOLOGIN`, `service_role NOLOGIN`
  — чтобы все `GRANT`-ы из 70 миграций легли как есть;
- login-роль **`app_user`** `IN ROLE authenticated` (наследует гранты, попадает
  под политики `TO authenticated`, НЕ владелец таблиц → RLS применяется);
- admin-путь = owner-роль Neon (владелец таблиц обходит RLS — аналог service_role);
- схема `auth`: таблица `auth.users` (id uuid PK, email + unique lower(email),
  encrypted_password, failed_attempts, locked_until, created_at…; таблица
  сессий НЕ нужна — ревью V2), функция `auth.uid()` =
  `nullif(current_setting('app.user_id', true), '')::uuid` — обязательно
  **STABLE** (иначе initplan-оптимизация RLS из 20260611101200 молча
  деградирует в построчный вызов — ревью V3);
- pgcrypto ставится **`WITH SCHEMA extensions`** (вызовы в
  user_login_secrets схемо-квалифицированы: `extensions.pgp_sym_encrypt` и
  т.д.) — схему `extensions` при чистке слепка НЕ сносить (ревью V3);
- **ACL-аудит после прогона слепка (ревью V3-3):** известная грабля проекта —
  `db reset` терял DML-гранты (memory: reset grants). Явный чек-скрипт:
  гранты по каждой таблице для `authenticated`, КОЛОНОЧНЫЕ гранты `users`
  (salary_* закрыты, остальное открыто), схемы `private`/`auth` недоступны
  `app_user`. Blanket-GRANT запрещён (вскрыл бы salary_*). Чек входит в
  гейт сессии 1.

Каждый пользовательский запрос идёт через обёртку
`userDb(userId, tx => …)` = interactive-транзакция Prisma:
`set_config('app.user_id', $id, true)` → запросы. Забыли обёртку →
`auth.uid()` NULL → RLS отрезает всё → **пустой результат, а не утечка**
(fail-closed по построению).

### 4.2 Свой auth (замена GoTrue)

> **Ревью V2 (внешний голос; ЗАМЕНЯЕТ решение A1).** Ротация refresh-токенов
> и таблица сессий — паттерн публичных сервисов; для внутренней CRM выбрана
> радикально более простая схема «скользящий JWT», при которой гонки
> параллельного обновления исчезают как класс (перевыпуск идемпотентен),
> а таблица `auth.sessions` не нужна вовсе.

- Логин: `bcryptjs.compare` с перенесёнными хешами → ОДНА httpOnly-кука
  с JWT (HS256, секрет `AUTH_SECRET`; claims: `sub`, `email`, `pwd_version`,
  `iat`; exp = 30 дней бездействия).
- **Скользящее продление:** `proxy.ts` проверяет подпись локально
  (jose.verify, без БД) и, если токену > 24 ч, молча перевыпускает куку
  (тоже локально). Сотрудник вошёл раз — работает месяцами, как при Supabase.
  Абсолютный потолок жизни без перелогина — 90 дней (по `iat` первичного входа).
- **Инвалидация без таблицы сессий:** `pwd_version` (int в `public.users`,
  инкремент при каждой смене/выдаче пароля) сравнивается с клеймом в
  `getCurrentUser` — тем же единственным запросом профиля, что уже есть.
  Смена пароля мгновенно сбрасывает ВСЕ устройства; деактивация
  (`is_active=false`) мгновенно режет доступ (проверка на каждом рендере,
  как сейчас). Кнопка «выйти со всех устройств» = тот же инкремент.
- **Rate-limit логина (ревью V3-4):** GoTrue давал его из коробки — в свой
  логин добавляем счётчик неудач + `locked_until` в `auth.users`
  (экспоненциальная задержка после N неудач). Временные пароли — 6 символов,
  перебор без лимита реалистичен.
- **Урок текущего proxy (ревью V3):** пересоздание response в middleware
  роняет тело POST server-action (`src/proxy.ts:30-37`, логин уже ломался
  об это) — новый proxy НЕ трогает request на POST к публичным путям;
  требование фиксируется тестом.
- `getCurrentUser`: verify JWT без сети → профиль из `public.users`
  (единственный БД-запрос — как сейчас; там же is_active, pwd_version, caps).
- Управление доступами владельцем (`/settings/users`): create/update/delete
  переписываются с `auth.admin.*` на прямые операции admin-пула с
  `auth.users`; RPC `get/set_user_login_secret`, `user_delete_blockers`
  остаются как есть.
- Email-флоу (приглашение/сброс пароля) — свой: таблица
  `auth.password_reset_tokens` (одноразовый токен, TTL), письмо через
  nodemailer/SMTP, страница установки пароля заменяет `/auth/confirm`.
- SQL-функции user management, лезущие в `auth.users`
  (миграции 20260530150000, 20260630120000) — адаптировать под нашу
  `auth.users` (проверить используемые поля).

### 4.3 Слой данных

- `prisma db pull` по готовой схеме → schema.prisma → типизированный клиент.
- **Ревью A3:** колонки `users.salary_mode` / `salary_fixed_amount` в
  schema.prisma помечаются `@ignore` (column-level привилегии БД запрещают
  их обычным запросам; Prisma по умолчанию тянет все колонки — без @ignore
  каждый users-запрос под app_user падал бы). Чтение/запись ЗП — только через
  DEFINER-RPC (`payroll_employee_summary`, `manage_user_salaries`), как сейчас.
  ⚠ Это правило распространяется на будущие приватные колонки users.
- Два пула: user (`app_user`, через `userDb`-обёртку) и admin (owner,
  `adminDb` — только 7 machine-роутов, seed, тест-фикстуры; зеркало
  текущего правила «service_role только системно», CLAUDE.md §2).
- **Ревью Q1:** правило «adminDb только системно» закрепляется механически —
  ESLint `no-restricted-imports`: импорт admin-клиента разрешён ТОЛЬКО в
  явном allowlist (cron/telegram/calendar/OnlyOffice-роуты, users-admin
  экшены, scripts/, tests/helpers). Настраивается в сессии 1, до начала
  массового переписывания.
- PostgREST-паттерны → Prisma: embedded-join `client:client_id(...)` →
  `include/select`; `.or(...)` строки → `OR:[...]`; `.range()` → skip/take;
  `{count:'exact', head:true}` → `count()`.
- **Ревью Q2:** RPC-вызовы НЕ раскиданы по файлам, а собраны в
  `lib/db/rpc.ts` — типизированный реестр всех ~26 SQL-функций (одна
  обёртка на функцию, параметры и результат типизированы). Рядом
  `lib/db/errors.ts` — единый маппер ошибок Prisma/Postgres (P2002 unique,
  P2003 FK, RLS-отказы, check-violations) в человеческие сообщения форм
  (стыкуется с `validation.ts` из v3). Оба модуля — сессия 1.
- **Механика userDb — спроектирована, а не «одной строкой» (ревью, внешний
  голос P1-4):** транзакция оборачивает ОДИН запрос/одну query-функцию
  (batch `$transaction([set_config, query])` — официальный паттерн Prisma),
  НЕ весь рендер страницы: иначе сериализуются запросы, ломается
  `Promise.all`-параллелизм из v3 s4 и пробивается 5-секундный дефолт
  interactive-tx (P2028 на холодном Neon). Параллельные транзакции из одного
  рендера разрешены; timeout поднят явно; connection string — Neon pooled
  с отключённым кэшем prepared statements; connection_limit выверяется под
  serverless. Мульти-шаговые записи (actions) — одна interactive-tx на action.
- Схему `private` в Prisma-модель НЕ вводим (доступ только через RPC-реестр);
  `auth.users` — через `$queryRaw` admin-пула (ревью V3).
- **После первого `prisma db pull` схема ведётся руками** — повторный pull
  сотрёт ручные `@ignore` (A3); правило фиксируется комментарием в
  schema.prisma (ревью V3).

### 4.4 Файлы

- Интерфейс `lib/storage.ts` (4 операции), реализация R2 (или Vercel Blob).
- Convention `storage_key` (`cases/<case_id>/<uuid>--<slug>`) сохраняется —
  в БД ничего менять не надо.
- Доступ к файлам и так контролируется API-роутами/экшенами с проверкой прав
  (превью/скачивание через подписанные URL, OnlyOffice — свои JWT);

- Скрипт переноса: список из `documents.storage_key` → download из Supabase →
  upload в R2 → сверка размеров/количества.

### Деплой и git на время цикла (ревью V1; УТОЧНЕНО владельцем в сессии 1, 2026-07-14)
Работаем в **master** (одна ветка, привычный флоу). Автодеплой Vercel
**НЕ трогаем** — вместо этого жёсткое правило: **`git push` ЗАПРЕЩЁН до
сессии 7**. Пуши делает только исполнитель и только по явной команде
владельца, поэтому случайный пуш исключён; весь цикл копится локально,
в сессии 7 — один push всего стека после переключения env.
Хотфикс прода при необходимости: ветка от `origin/master` (= прод-код) →
push ТОЛЬКО этой ветки → ручной deploy ветки в дашборде Vercel
(master не пушить).
⚠ Риск-компенсация: прод в это время НЕ обновляется — все изменения
копятся до вечера переезда.

## 12. Параллелизация работ

Последовательная реализация: 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8 — каждая сессия
опирается на предыдущую (фундамент → auth → данные → файлы → чистка → прод).
Единственная безопасная параллель: модуль `lib/storage.ts` + R2-аккаунт
(сессия 5) можно готовить параллельно с сессиями 3–4 — модуль не пересекается
с переписыванием запросов; сама замена вызовов файлов — после. Выигрыш мал,
параллельные ворктри не оправданы.
