# Стартовый промт для Claude Code

> Скопируй этот текст и отправь в Claude Code в корне нового (пустого) репозитория,
> положив рядом файл `CLAUDE.md`. Предполагается, что gstack уже установлен.

---

Мы строим CRM-систему для юридической компании. Полное описание — стек, доменная
модель, роли, правила доступа, этапы, фазы, **дизайн (раздел 11)** и **рабочий процесс
gstack (секция «gstack»)** — лежит в `CLAUDE.md` в корне репозитория. **Сначала прочитай
его целиком**, держи как источник правды и не отклоняйся от бизнес-правил (раздел 7) без
моего подтверждения.

Сейчас делаем только **Phase 1 (MVP)** из раздела 8. Не пытайся сделать всё сразу.
Идём по шагам: перед каждым крупным шагом — короткий план (что создаёшь, какие файлы,
какие решения) и жди моего «ок». После шага — короткое резюме и как проверить.

## Как работаем (применять на каждом шаге)

- **gstack по максимуму, проактивно.** Сам определяй нужный скил по задаче и вызывай его,
  не дожидаясь слеша. Следуй спринт-циклу из секции gstack: Think → Plan → Build → Review
  → Test → Ship → Reflect.
- **Перед кодом фичи** — планирование (`/plan-eng-review` для архитектуры).
- **После кода** — `/review`; для всего, что касается прав доступа (RLS), хранения
  документов и аутентификации — обязательно ещё `/cso` (безопасность) и второе мнение `/codex`.
- **Для каждого UI-экрана** — `/design-review` (полировка) + `/qa` (поведение в реальном
  браузере через `/browse`). Дизайн строго по разделу 11 CLAUDE.md.
- Безопасность доступа — приоритет: никаких обходов RLS ради скорости.
- Типобезопасно, без лишнего `any`. Веб-браузинг — только через `/browse`.

## Шаги

**Шаг 0 — Инициализация и gstack.**
1) Проверь, что gstack активен и скилы распознаются; включи проактивный режим:
   `~/.claude/skills/gstack/bin/gstack-config set proactive true`.
2) Создай проект на стеке из CLAUDE.md (Next.js App Router + TypeScript, Supabase,
   Tailwind + shadcn/ui). Структура папок, линтер, `.env.example`, локальный Supabase.
3) Обнови раздел «Команды» в CLAUDE.md реальными командами.

**Шаг 1 — Схема БД + RLS.**
SQL-миграции для всех таблиц из раздела 5 (`users`, `clients`, `cases`, `documents`,
`tasks`, `payments`, `activity_log`). Enum для ролей, типов дел и этапов. Настрой **RLS**
строго по матрице доступа и правилам из раздела 4. Перед применением прогони `/cso` и
покажи мне политики RLS текстом — я проверю логику доступа. Сид с тестовыми
пользователями всех 4 ролей (через `service_role`).

**Шаг 2 — Auth и роли.**
Вход через Supabase Auth, роль текущего пользователя, ролевые маршруты, запрет доступа
к чужим разделам. Убедись, что запросы идут с сессией пользователя (RLS работает),
а не через `service_role`. После — `/review` + `/cso`.

**Шаг 3 — Визуальная система (до UI-экранов).**
Запусти `/design-consultation` и зафиксируй визуальную систему по разделу 11: светлая
и тёмная темы, пара шрифтов (НЕ Inter/Roboto/Arial), акцентный цвет, дизайн-токены
(CSS-переменные), табличные цифры, базовые компоненты поверх shadcn/ui (кастомизированные,
не дефолтные), командная палитра Cmd/Ctrl-K, заготовки состояний (empty / skeleton / error).
Цель — уровень Linear / Vercel / Stripe Dashboard. Покажи мне визуальную систему до того,
как строить экраны.

**Шаг 4 — Клиенты.** Список + карточка + создание/редактирование; на карточке клиента —
все его дела. После — `/design-review` + `/qa`.

**Шаг 5 — Дела и карточка дела.** Список с фильтрами (статус, тип, клиент, ответственный)
и карточка дела со всеми полями раздела 5 (данные · документы · задачи · финансы · история).
Сильные таблицы (липкие заголовки, сортировка, числа вправо). После — `/design-review` + `/qa`.

**Шаг 6 — Этапы.** Воронка (доска/список), смена этапа **только вперёд** (правило 2,
раздел 7); возврат назад — только owner/admin с записью в `activity_log`. После — `/qa`.

**Шаг 7 — Задачи и календарь.** Задачи по делу (любой специалист — себе и коллегам),
дедлайны, заседания, общий календарь. После — `/design-review` + `/qa`.

**Шаг 8 — Документы.** Загрузка/хранение/скачивание по делу в приватный бакет Supabase
Storage с доступом по правилам дела. Прогони `/cso` (хранение и доступ к файлам).

**Шаг 9 — Финансы (базово).** Сумма по договору и платежи, автоматический расчёт
оплачено/долг. Без инвойсов и времени (это Phase 2).

**Шаг 10 — Поиск, журнал, напоминания.** Поиск по делам/клиентам/документам; запись
изменений в `activity_log`; напоминания о приближающихся сроках в интерфейсе. После — `/qa`.

## По завершении фич
- Выкатка → `/ship` или `/land-and-deploy`, пост-деплой — `/canary`.
- Документация → `/document-release` / `/document-generate`.
- Ретро → `/retro`, уроки → `/learn`.

## Чего НЕ делать
- Не реализуй Phase 2/3 (шаблоны, инвойсы, время-трекинг, клиентский портал, э-подпись,
  интеграции, миграция) — только оставляй задел под них.
- Если требование неоднозначно или расходится с CLAUDE.md — остановись и спроси.

Начни с Шага 0: проверь gstack, включи проактивный режим, покажи план инициализации
и список файлов — и жди «ок».
